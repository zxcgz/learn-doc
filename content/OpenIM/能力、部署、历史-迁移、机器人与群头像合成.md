> 面向工程团队的 OpenIM 实操笔记：从“一键起服与快速体验”，到“历史消息拉取/迁移”、消息机器人落地、以及“服务端合成九宫格群头像”的完整方案。

## 1）最短路径起服与测试

**目标**：拉起一套可用的 OpenIM 基础栈 + Web 测试端。

**步骤**

1. 使用官方 Compose 编排或 Helm Chart，准备依赖（MongoDB / Redis / Kafka+ZK / MinIO 等）。
2. 启动 `openim-docker`（或等价部署）。
3. 浏览器访问 `http://<服务器IP>:11001`（PC Web 测试端），按注册流程进入对话。
4. 管理后台可用于用户/群管理、回调配置与排障（端口依据你的编排）。

**小规格建议**

- 1 vCPU / 4 GB 可跑核心 IM 并做小规模验证；需要音视频（OpenMeeting/RTC + TURN）时，建议将 RTC 与 TURN 拆到另一台小实例或升级至 2C/8G+。

## 2）历史消息：拉取与漫游

- 客户端通过 SDK 的**分页历史**接口按会话维度向前翻页；新设备重装后靠**漫游/云端历史**补齐。
- 典型查询方式：指定锚点消息（或序号/时间戳）+ 拉取条数，实现“无限上拉”。

**边界处理**

- 离线期间消息补拉与本地去重：依据会话同步光标（maxSeq/readSeq）实现幂等恢复。
- 富媒体历史：客户端拉取时异步加载缩略图/原图，失败重试与占位图策略需统一。

## 3）消息迁移：从他家 IM 导入到 OpenIM

**总体链路**

1. **导入账号/关系/群**：将外部用户、好友关系、群资料批量导入。
2. **导入历史消息**：使用“管理员 REST 的模拟发送/写入接口”按原时间线写入历史。
3. **恢复会话状态**：同步置顶、免打扰、未读数（如业务上需要对齐）。
4. 客户端侧直接**拉取历史**与增量同步，用户无感迁移。

**富媒体导入**

- 先将图片/语音/视频/文件上传到你的对象存储（S3/OSS/MinIO），将可访问的 URL 写入消息体相应字段即可。

## 4）消息机器人：单聊与群聊

**两条落地路径**

- **Webhook + 管理 REST（推荐）**
  - 开启“发送后/前回调”；当用户给机器人发消息，OpenIM 将事件 POST 到你的回调服务。
  - 回调服务调用“发送消息 REST 接口”，以机器人账号的 `userID` 回发响应。
  - 优点：无需让机器人长连在线，服务端易于与 LLM/业务系统整合。
- **长连机器人**
  - 为 `bot_xxx` 签发用户 token，机器人进程使用 SDK/自研长连登录网关，像普通用户一样收发。
  - 适合需要实时订阅、状态保持、复杂交互的场景。

**群机器人**

- 将机器人加入群聊；回调订阅群内消息事件；以 `sessionType=3` + `groupID` 回复即可。

**身份建模**

- OpenIM 没有“bot 类型”，本质是普通用户。推荐在 `user.ex` 或你的用户库标 `is_bot=true`、`scopes=[...]`；前端显示“BOT”徽标，服务端回调做白名单、限流与权限校验。

## 5）头像：个人头像与群聊九宫格（服务端合成）

**个人头像**

- 直接设置 `faceURL`（URL 文本）；OpenIM 不会自动抓取或镜像图片。建议使用 HTTPS CDN。

**群头像（推荐九宫格合成）**

- 服务器端没有“默认群头像”。可实现一个 **GroupAvatar 后端**：
  1. 触发：群创建/成员变更/成员头像变更。
  2. 拉取群成员头像 URL（取前 N 个，如 9）。
  3. 并发下载、圆形裁剪，按 3×3 布局合成。
  4. 上传对象存储得出 CDN URL。
  5. 调用“更新群资料”将 `groupFaceURL` 写为该合成图 URL。

**实现要点**

- 事件驱动 + 去重队列（`groupID + 成员头像哈希`）；
- 稳定排序（如按入群时间或稳定哈希）避免频繁抖动；
- 下载加超时/尺寸限制，失败使用占位图；
- 文件名使用内容哈希，命中即复用；回写失败重试退避。

**合规提示**

- 全流程仅使用 OpenIM Server 的公开 API 与宽松许可图像库；不复用 AGPL SDK 代码，自然不受 AGPL 传导影响。

## 6）常见问题与排障

- **消息丢失/重复**：检查本地库去重策略、会话同步光标、ACK/重发逻辑。
- **通知/推送迟滞**：确保客户端正确上报推送 token（FCM/JPush/个推均可），服务端通道配置正确。
- **迁移后对齐**：历史导入与会话状态恢复顺序正确；客户端初次登录完成漫游后再核对未读。
