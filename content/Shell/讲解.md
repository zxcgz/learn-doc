# 0. 使用说明
- 这是一套可直接用于对外讲解的可视化图集（Mermaid）。
- 覆盖：系统总览、线上/线下授权流程、打包与分发、N-of-M 审批解封模型、版本与升级、运行期管控与拉闸、里程碑甘特图。
- 讲解顺序建议：1→2→3→4→5→6→7。

# 1) 系统总览架构图（人/机边界与信任根）
```mermaid
flowchart LR
  subgraph People[管理员/运维]
    U1[用户名/密码 登录主站控制台]
    P1[Passkeys仅高危审批/下证使用]
  end

  subgraph Controller[主站（Controller）]
    GW[API网关mTLS+证书策略判权]
    CA[Issuing CA\n授权证书签发]
    POL[许可表/配额/限流]
    PKG[打包调度器GitHubActionsOIDC]
    AR[制品库/签名清单/SBOM]
    STATS[统计与审计]
  end

  subgraph Agents[子站（Agents）]
    A1[子站-公网\n自动注册]
    A2[子站-内网\n线下授权]
  end

  subgraph Roots[离线信任根]
    ROOT[RootCA离线]
  end

  U1 -->|用户名/密码| GW
  P1 -->|WebAuthn 审批| CA
  A1 <-->|mTLS| GW
  A2 <-->|线下包| GW

  CA -->|授权证书| A1
  CA -->|授权证书离线| A2

  PKG -->|OIDC| GH[(GitHub Actions)]
  GH -->|构建产物| AR
  AR -->|在线分发/离线包| A1
  AR -->|离线导入| A2

  POL --> GW
  STATS --- GW
  ROOT -.签发中级.-> CA
```
**要点**

- 登录统一用用户名/密码；**Passkeys 只参与高危审批/下证**。
- 子站功能受**授权证书**与**API 网关策略**双重约束。
- 打包在主站统一调度，制品可在线下发或做**离线包**导入。

# 2) 线上注册与授权流程（公网子站）
```mermaid
sequenceDiagram
  participant Agent as 子站（公网）
  participant Controller as 主站
  participant Admin as 管理员（Passkeys 审批）
  participant CA as Issuing CA

  Agent->>Controller: 注册请求 (CSR + 站点信息)
  Controller-->>Agent: 进入“未授权子站”列表
  Admin->>Controller: 控制台审批（用户名/密码 登录）
  Admin->>Controller: 高危动作需 Passkeys (WebAuthn)
  Controller->>CA: 发起签发授权证书
  CA-->>Controller: 授权证书 (含角色/配额/版本/过期策略)
  Controller-->>Agent: 在线下发授权证书
  Agent->>Controller: mTLS 带证访问，启用功能
```
**要点**
- 未授权子站只可上报信息；审批后才解锁功能。
- 授权证书是“硬许可证”，携带功能位/配额/版本门槛。

# 3) 内网线下授权流程（完全离线）
```mermaid
sequenceDiagram
  participant Agent as 子站（内网）
  participant USB as 线下介质
  participant Controller as 主站
  participant CA as Issuing CA

  Agent-->>USB: 导出“注册包”(CSR + 站点指纹)
  USB-->>Controller: 人工带到主站
  Controller->>CA: 审批(必要时 Passkeys)后签发
  CA-->>Controller: 授权证书
  Controller-->>USB: 生成“授权离线包”\n(证书+策略清单，签名 + 用子站公钥加密)
  USB-->>Agent: 离线导入，启用功能
```
**要点**
- 离线包双重保护：**主站签名清单 + 子站公钥加密**，防篡改与错站导入。

# 4) 打包与分发流水线（含限流/去重）
```mermaid
flowchart TB
  subgraph Agent[子站]
    RQ[发起打包请求参数集]
  end
  subgraph Controller[主站]
    POL[配额/令牌桶/并发池]
    DEDUP[去重缓存参数→制品]
    QUEUE[构建队列]
    OIDC[OIDC 到 GitHub]
    AR[制品库 + 哈希/签名清单/SBOM]
  end
  GH[(GitHub Actions)]

  RQ -->|mTLS & 证书判权| POL
  POL -->|允许| DEDUP
  DEDUP -->|命中| AR
  DEDUP -->|未命中| QUEUE --> OIDC --> GH --> AR
  AR -->|在线推送| Agent
  AR -->|离线导出| USB[(离线介质)] --> Agent
```
**要点**
- **三道闸**：配额/并发、去重缓存、人工审批闸（超过阈值触发）。
- 制品全量存证（哈希/签名清单/SBOM），可复用、可追溯。

# 5) 版本与升级策略（主站≥子站）
```mermaid
stateDiagram-v2
  [*] --> Running
  state Running {
    [*] --> Compatible: 主站版本 ≥ 子站下限
    Compatible --> UpdatePrompt: 主站发布新版本 & 证书策略允许提示
    UpdatePrompt --> AutoUpdate: 证书策略=自动升级
    UpdatePrompt --> ManualUpdate: 证书策略=人工确认
    AutoUpdate --> Compatible
    ManualUpdate --> Compatible
    Compatible --> Incompatible: 主站升级后不再满足下限
  }
  Incompatible: 子站提示必须升级\n(在线拉取或离线包导入)
  Incompatible --> Compatible: 升级完成 & 自检通过
```
**要点**
- 版本门槛来自授权证书字段（如 `version_floor`）。
- 在线/离线两路升级；失败可回滚。

# 6) N-of-M 审批解封模型（凭据≠份额）
```mermaid
flowchart LR
  KEK{{KEK 解封密钥}}
  subgraph Shares[Shamir份额-以管理员为单位]
    S1[Share_A]
    S2[Share_B]
    S3[Share_C]
  end
  subgraph CredsA[管理员A的多把凭据]
    A1[PasskeyA1]
    A2[FIDO2 KeyA2]
  end
  subgraph CredsB[管理员B的多把凭据]
    B1[PasskeyB1]
  end
  subgraph CredsC[管理员C的多把凭据]
    C1[PasskeyC1]
    C2[FIDO2 KeyC2]
  end

  A1-- 解包 -->S1
  A2-- 解包 -->S1
  B1-- 解包 -->S2
  C1-- 解包 -->S3
  C2-- 解包 -->S3

  S1 & S2 -. 2-of-3 .-> KEK
  KEK -->|解封| EncCA[(Enc-CA私钥)]
```
**要点**
- 阈值按**份额/管理员**计（如 2-of-3），**不是凭据数量**。
- 同一份额可被多把凭据“包裹”，丢一把凭据≠丢份额。
- 解封仅在下证/高危动作瞬时进行，用毕即抹除。

# 7) 运行期管控与“拉闸”
```mermaid
flowchart LR
  subgraph Controller[主站]
    CRL[CRL/ΔCRL 发布]
    AL[许可表/epoch 策略]
  end
  subgraph Agents[子站]
    V1[握手校验证书+CRL]
    V2[接口授权许可表-epoch]
  end

  CRL --> V1
  AL --> V2


```
**要点**
- 证书长效不代表不可控：**CRL/ΔCRL**（握手前） + **许可表/epoch**（握手后）双层控制。

## 讲解小抄（每张图的“一句话”）
- **总览**：主站是策略与签发核心，子站受证书和网关双控；打包统一在主站调度。
- **线上授权**：公网子站自动注册，Passkeys 审批后在线下发证书。
- **线下授权**：注册包→授权离线包（签名+加密）→导入启用。
- **打包流水线**：令牌桶+去重+Actions+制品库，产物可在线/离线分发。
- **版本**：主站≥子站；是否升级由证书策略决定。
- **N-of-M**：凭据是钥匙，份额是票；2-of-3 解封 KEK，私钥仅瞬时在内存。
- **拉闸**：长效证书 + CRL/ΔCRL + 许可表，实现分钟级停/放行。
