## 快速总览（TL;DR）

- **先看峰值**：`Max_used_connections`，别盲目拉太高。
    
- **临时改**：`SET GLOBAL max_connections = 800;`
    
- **持久化**（MySQL 8.0+）：`SET PERSIST max_connections = 800;`
    
- **或配置文件**：`[mysqld]\nmax_connections = 800` + 重启服务。
    
- **配套**：开启 `skip-name-resolve`、提高 `open_files_limit`、调小 `wait_timeout`。
    
- **估算**：每连接消耗 0.5–2+ MB（视负载而定），给系统与批处理留 20% 余量。
    
- **应用侧**：各服务连接池上限**合计** ≤ `max_connections × 0.8`。

## 观察当前与历史峰值

```sql
SHOW VARIABLES LIKE 'max_connections';
SHOW STATUS LIKE 'Threads_connected';
SHOW STATUS LIKE 'Max_used_connections';
```

> **建议**：以 `Max_used_connections` 为基准，结合业务增长给出弹性余量，而非成倍盲升。

## 临时提高（立即生效）

```sql
-- 需要 CONNECTION_ADMIN（MySQL 8.0+）或 SUPER 权限
SET GLOBAL max_connections = 800;  -- 按需调整
```

> 重启后失效。适合应急或压测。

---

## 持久化提高（重启后保留）

### MySQL 8.0+（SET PERSIST）

```sql
SET PERSIST max_connections = 800;  -- 写入 mysqld-auto.cnf
```

### 修改配置文件（所有版本通用）

常见路径：

- Debian/Ubuntu：`/etc/mysql/mysql.conf.d/mysqld.cnf`
    
- CentOS/RHEL：`/etc/my.cnf` 或 `/etc/my.cnf.d/server.cnf`
    

```ini
[mysqld]
max_connections = 800
```

重启服务：

```bash
# 依环境不同可能是 mysql / mysqld / mariadb
sudo systemctl restart mysql
```

## Docker 部署的两种改法

**方式 A：启动参数**

```yaml
services:
  db:
    image: mysql:8.0
    command: ["mysqld","--max_connections=800"]
    # 其他环境变量/卷/端口等按需配置
```

**方式 B：挂载自定义配置**

```yaml
services:
  db:
    image: mysql:8.0
    volumes:
      - ./my.cnf:/etc/mysql/conf.d/99-custom.cnf:ro
```

`./my.cnf` 内容：

```ini
[mysqld]
max_connections = 800
```

## 配套与资源上限

### mysqld 常用配套参数

```ini
[mysqld]
# 连接与超时
max_connections = 800
wait_timeout = 600              # 非交互式空闲超时（秒）
interactive_timeout = 600       # 交互式空闲超时（秒）

# 握手与解析（强烈建议）
skip-name-resolve               # 关闭反向 DNS，用户授权请用 IP/网段

# 线程与缓存
thread_cache_size = 64
table_open_cache = 4096
table_open_cache_instances = 16

# 文件句柄上限（需系统层支持）
open_files_limit = 65535
```

> [!warning] 授权注意  
> 开启 `skip-name-resolve` 后，**不要**再使用 `'user'@'hostname'` 形式授权，改用 `'user'@'%'` 或 `'user'@'10.0.%'` 等 **IP/网段**。

### systemd 文件句柄上限

```bash
sudo systemctl edit mysql
# 写入以下内容后保存退出
# [Service]
# LimitNOFILE=65535

sudo systemctl daemon-reload
sudo systemctl restart mysql
```

> 不提高 systemd 的 `LimitNOFILE`，`open_files_limit` 往往抬不起来。

## 安全上限与内存估算

- **每连接内存**（粗略）：`0.5–2+ MB/连接`（轻负载），复杂排序/联接时更高（会临时申请大缓冲）。
    
- **经验**：让 `max_connections` 带来的总连接开销≈**物理内存的 10–20%**；同时给 InnoDB Buffer Pool、查询缓冲、OS Cache、备份任务等留足空间。
    
- **应用总和**：各服务连接池上限**合计** ≤ `max_connections × 0.8`，给运维/批处理/临时会话≥20% 余量。
    
- **HikariCP 常见基线**：`maximumPoolSize ≈ CPU 核数 × (2~4)`，别盲目配到 50~100。

## 验证与回归检查

```sql
-- 确认生效
SHOW VARIABLES LIKE 'max_connections';

-- 观察是否仍撞上限
SHOW STATUS LIKE 'Max_used_connections';
SHOW STATUS LIKE 'Connection_errors_max_connections';

-- 观察空闲连接
SHOW STATUS LIKE 'Threads_connected';
```

> 在业务高峰或压测时复核数据；若 `Max_used_connections` 长期远低于上限，可回调上限值节省内存。

## 实战贴士（避免新问题）

- **只升上限≠治本**：配合较短的 `wait_timeout`，减少长时间 `Sleep` 占坑。
    
- **关闭反查**：`skip-name-resolve` 能显著减少连接握手延迟与偶发卡顿。
    
- **连接池与 DB 对齐**：代理层（如 ProxySQL）的前端/后端 `max_connections` 也要匹配 DB，并预留 20–30% 余量。
    
- **权限账号**：准备一个拥有 `CONNECTION_ADMIN`（或旧版 `SUPER`）的账号，必要时可在“超过上限”时仍能接入做处置。
    
- **渐进式变更**：分阶段提高，观察内存曲线与 `Threads_connected`，避免一次性跃迁引发 OOM 风险。

## 整套模板与片段库

### 1) `my.cnf` 基础模板

```ini
[mysqld]
# —— 核心 —— 
max_connections = 800

# —— 超时 —— 
wait_timeout = 600
interactive_timeout = 600

# —— 解析/握手 —— 
skip-name-resolve

# —— 线程/缓存 —— 
thread_cache_size = 64
table_open_cache = 4096
table_open_cache_instances = 16

# —— 句柄上限（需 systemd 配合） —— 
open_files_limit = 65535
```

### 2) systemd 覆盖（提高 NOFILE）

```ini
# /etc/systemd/system/mysql.service.d/override.conf
[Service]
LimitNOFILE=65535
```

> [!tip] 迁移/扩容小建议  
> 准备“专用运维连接池”（极小上限，短超时）给运维任务与告警系统使用，避免业务洪峰把连接挤爆后无法登陆排障。

> [!note] 回退策略  
> 若升高后出现内存紧张或系统抖动，可先把 `max_connections` 回退 10–20%，同步缩小各应用池上限，再逐步调优 SQL 与索引以降低并发连接占用。
